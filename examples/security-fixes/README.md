# Security Remediation Examples

This directory contains practical examples of common security vulnerabilities and their automated fixes for Claude Code integration.

## Files

- `vulnerable-example.go` - Demonstrates vulnerable code patterns and their secure fixes
- `README.md` - This documentation file

## Purpose

These examples demonstrate how Claude Code can automatically remediate common security issues detected by our security scanning tools (Trivy, Nancy, gosec, staticcheck).

## How to Use These Examples

### 1. Generate Remediation Report

```bash
# Generate the security remediation report
make security-remediation-report

# View the full report
jq . /tmp/cfgms-security-remediation.json
```

### 2. Run Example Script

```bash
# Run the example remediation workflow
./scripts/example-remediation.sh
```

### 3. Test Security Scanning

```bash
# Run security scan on the example file
gosec ./examples/security-fixes/

# This should detect the vulnerable patterns in vulnerable-example.go
```

## Vulnerability Patterns Demonstrated

### 1. Integer Overflow (G115)
- **Issue**: Direct conversion from `int` to `uint32` without bounds checking
- **Fix**: Add bounds validation before type conversion
- **Code**: Lines 18-22 (vulnerable) vs 26-36 (secure)

### 2. Weak Random Number Generator (G404)
- **Issue**: Using `math/rand` for security-sensitive operations
- **Fix**: Use `crypto/rand` for cryptographically secure randomness
- **Code**: Lines 39-42 (vulnerable) vs 45-49 (secure)

### 3. TLS MinVersion Too Low (G402)
- **Issue**: TLS configuration allows insecure protocol versions
- **Fix**: Set MinVersion to TLS 1.2 or higher
- **Code**: Lines 52-57 (vulnerable) vs 60-65 (secure)

### 4. Subprocess with Variable (G204)
- **Issue**: Executing shell commands with user input
- **Fix**: Validate and sanitize input before execution
- **Code**: Lines 68-72 (vulnerable) vs 75-88 (secure)

### 5. File Inclusion via Variable (G304)
- **Issue**: Reading files with user-controlled paths
- **Fix**: Clean paths and validate against allowed directories
- **Code**: Lines 91-94 (vulnerable) vs 97-107 (secure)

### 6. Insecure File Permissions (G301, G302, G306)
- **Issue**: Creating files/directories with overly permissive permissions
- **Fix**: Use restrictive permissions (0750 for dirs, 0600 for files)
- **Code**: Lines 110-125 (vulnerable) vs 128-142 (secure)

### 7. HTTP Server Timeouts (G112)
- **Issue**: HTTP server without ReadHeaderTimeout (Slowloris vulnerability)
- **Fix**: Configure appropriate timeouts
- **Code**: Lines 145-151 (vulnerable) vs 154-162 (secure)

## Claude Code Integration

When Claude Code processes the remediation report, it can:

1. **Identify the pattern** using the gosec rule code (G115, G404, etc.)
2. **Locate the vulnerable code** using file path and line number
3. **Apply the appropriate fix** using patterns from this example file
4. **Validate the fix** by running the security scan again

### Example Workflow

```bash
# 1. Generate report
make security-remediation-report

# 2. Claude Code reads the report and identifies issues like:
# "G115 (HIGH): integer overflow conversion int -> uint32 at file.go:20"

# 3. Claude Code applies fix by:
# - Adding bounds checking before type conversion
# - Following the pattern from vulnerable-example.go lines 26-36

# 4. Validate the fix
make security-gosec
make test
```

## Files Generated by Security Tools

When you run the remediation report, these temporary files are created and analyzed:

- `/tmp/trivy-remediation.json` - Trivy vulnerability results
- `/tmp/nancy-remediation.json` - Nancy dependency scan results  
- `/tmp/gosec-remediation.json` - gosec security pattern results
- `/tmp/staticcheck-remediation.json` - staticcheck analysis results
- `/tmp/cfgms-security-remediation.json` - Combined remediation report
