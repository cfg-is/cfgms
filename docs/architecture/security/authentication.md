# Authentication

This document details the authentication mechanisms used in CFGMS.

## Overview

CFGMS implements a multi-layered authentication approach, ensuring that all components and users are properly authenticated before accessing any resources.

## Authentication Mechanisms

### 1. Internal Component Authentication

#### Mutual TLS (mTLS)

- **Purpose**: Secures communication between internal components (Controller, Steward, Outpost)
- **Implementation**:
  - All internal communication uses gRPC with mTLS
  - Each component has its own certificate signed by the Controller's CA
  - Certificates include component-specific identity information
  - Certificate validation includes checking against a Certificate Revocation List (CRL)
- **Certificate Lifecycle**:
  - Certificates are automatically generated during component registration
  - Certificates have a configurable validity period
  - Certificates can be rotated without service interruption
  - Revoked certificates are immediately rejected

#### Bootstrap Authentication

- **Purpose**: Secures the initial connection between components
- **Implementation**:
  - One-time bootstrap tokens for initial component registration
  - Tokens are short-lived and single-use
  - Tokens are securely distributed out-of-band
  - After successful registration, components switch to certificate-based authentication

### 2. External API Authentication

#### API Keys

- **Purpose**: Authenticates external clients accessing the REST API
- **Implementation**:
  - API keys are generated by the Controller
  - Keys can be scoped to specific resources or operations
  - Keys can have expiration dates
  - Keys can be revoked at any time
- **Key Management**:
  - Keys are stored securely (hashed)
  - Key rotation policies can be enforced
  - Key usage is audited
  - Keys can be associated with specific users or service accounts

#### OAuth 2.0 / OpenID Connect (Optional)

- **Purpose**: Provides integration with identity providers
- **Implementation**:
  - Support for standard OAuth 2.0 flows
  - Integration with common identity providers (Azure AD, Okta, etc.)
  - JWT validation for access tokens
  - Refresh token rotation

### 3. User Authentication

#### Local Authentication

- **Purpose**: Authenticates users directly against CFGMS
- **Implementation**:
  - Username/password authentication
  - Password policies enforced
  - Account lockout after failed attempts
  - Password reset functionality

#### External Identity Providers

- **Purpose**: Integrates with enterprise identity systems
- **Implementation**:
  - SAML 2.0 support
  - OpenID Connect support
  - LDAP/Active Directory integration
  - Attribute mapping for role assignment

## Authentication Flows

### Component Registration Flow

1. Component generates a bootstrap token request
2. Controller validates the bootstrap token
3. Controller generates a component-specific certificate
4. Component receives and installs the certificate
5. Component establishes secure communication using the certificate

### API Authentication Flow

1. Client obtains an API key from the Controller
2. Client includes the API key in all API requests
3. Controller validates the API key
4. Controller checks the key's permissions
5. Request proceeds if authorized

### User Authentication Flow

1. User provides credentials (username/password or SSO)
2. Controller validates credentials against the appropriate identity source
3. Controller generates a session token
4. User includes the session token in subsequent requests
5. Controller validates the session token for each request

## Best Practices

1. **Regular Rotation**
   - Certificates should be rotated regularly
   - API keys should have expiration dates
   - Passwords should be changed periodically

2. **Least Privilege**
   - Components should have minimal required permissions
   - API keys should be scoped to specific resources
   - Users should have role-based access

3. **Audit and Monitoring**
   - All authentication attempts should be logged
   - Failed authentication attempts should trigger alerts
   - Authentication patterns should be monitored for anomalies

4. **Secure Storage**
   - Credentials should never be stored in plain text
   - Secrets should be encrypted at rest
   - Access to credential storage should be restricted
