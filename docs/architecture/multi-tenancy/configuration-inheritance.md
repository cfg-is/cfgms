# Configuration Inheritance

## Overview

CFGMS implements a hierarchical configuration inheritance model where child tenants inherit configuration from their parent tenants. This inheritance is managed through a compilation process that merges configurations based on the tenant hierarchy and dynamic groups.

## Configuration Structure

CFGMS organizes configurations in a hierarchical structure:

1. **Base Configurations** - Applied to all endpoints, inherited through the tenant hierarchy
2. **Group Configurations** - Applied to endpoints matching dynamic group criteria
3. **Endpoint Configurations** - Applied to specific endpoints, not inherited

## Dynamic Groups

Dynamic groups are defined using DNA attributes with simple or complex matching criteria:

```yaml
# Group Definitions
groups:
  # Simple group based on role
  web_servers:
    public: true  # Whether this group is inherited to child tenants
    match:
      dna.role: "web-server"
  
  # Complex group with multiple criteria
  high_performance_servers:
    public: true
    match:
      dna.cpu_cores: ">= 8"
      dna.memory_gb: ">= 32"
      dna.os: "ubuntu"
```

Groups can be marked as:

- **Public**: Inherited to child tenants
- **Private**: Not inherited, only applies within the defining tenant

## Group Inheritance

Groups are inherited through the tenant hierarchy:

1. Each tenant has a `groups.cfg` file defining its groups
2. Each tenant has an `inherited_groups.cfg` file containing groups inherited from parent tenants
3. When a child tenant is created or updated, its `inherited_groups.cfg` is generated by:
   - Reading the parent tenant's `groups.cfg`
   - Filtering to include only public groups
   - Merging with any existing inherited groups from the parent's `inherited_groups.cfg`

## Inheritance Rules

1. Base configurations are inherited from parent tenants
2. Public groups are inherited from parent tenants
3. Endpoint configurations are not inherited
4. Inheritance is depth-first (deepest override wins)
5. Arrays are merged based on key fields
6. Inheritance is explicit and traceable
7. Compilation happens at runtime

## Configuration Examples

### Base Configuration

```yaml
# /cfgms/tenants/acme/base/base.yaml
base:
  security:
    password_policy:
      min_length: 12
      require_special: true
      require_numbers: true
    mfa:
      enabled: true
      methods:
        - type: "totp"
          issuer: "ACME Corp"
  
  monitoring:
    intervals:
      health_check: "5m"
      config_check: "15m"
    alerts:
      channels:
        - type: "email"
          address: "ops@acme.com"
```

### Group Configuration

```yaml
# /cfgms/tenants/acme/groups/web_servers.yaml
group:
  name: "web_servers"
  public: true
  match:
    dna.role: "web-server"
  config:
    packages:
      - name: nginx
        state: present
    services:
      - name: nginx
        state: running
        enabled: true
    files:
      - path: /etc/nginx/nginx.conf
        content: |
          # Nginx configuration
        mode: "0644"
        owner: nginx
        group: nginx
```

### Child Configuration


### Group Definitions

```yaml
# /cfgms/tenants/acme/groups.cfg
groups:
  web_servers:
    public: true
    match:
      dna.role: "web-server"
  
  database_servers:
    public: true
    match:
      dna.role: "database-server"
  
  internal_servers:
    public: false  # Private group, not inherited
    match:
      dna.environment: "internal"
```

### Inherited Groups

```yaml
# /cfgms/tenants/bravo/inherited_groups.cfg
# This file is automatically generated from parent tenant's groups.cfg

    public: true

      dna.role: "web-server"
```

## Compilation Process

### Implementation

```go
type ConfigCompiler struct {
    tenantManager *TenantManager
    groupManager  *GroupManager
    configStore   *ConfigStore
}

type CompiledConfig struct {
    TenantID    string
    ParentID    string
    Config      *Config
    Groups      []string
    Timestamp   time.Time
}

func (cc *ConfigCompiler) CompileTenantConfig(tenantID string) (*CompiledConfig, error) {
    // Get tenant hierarchy
    hierarchy, err := cc.tenantManager.GetTenantHierarchy(tenantID)
    if err != nil {
        return nil, fmt.Errorf("failed to get tenant hierarchy: %w", err)
    }
    
    // Start with root config
    config := &Config{}
    
    // Apply each level's base config
    for _, tid := range hierarchy {
        baseConfig, err := cc.configStore.GetBaseConfig(tid)
        if err != nil {
            return nil, fmt.Errorf("failed to get base config for tenant %s: %w", tid, err)
        }
        
        // Merge configs
        if err := config.Merge(baseConfig); err != nil {
            return nil, fmt.Errorf("failed to merge base config for tenant %s: %w", tid, err)
        }
    }
    
    // Get groups for this tenant
    groups, err := cc.groupManager.GetTenantGroups(tenantID)
    if err != nil {
        return nil, fmt.Errorf("failed to get groups for tenant %s: %w", tenantID, err)
    }
    
    return &CompiledConfig{
        TenantID:  tenantID,
        ParentID:  hierarchy[len(hierarchy)-2], // Second to last is parent
        Config:    config,
        Groups:    groups,
        Timestamp: time.Now(),
    }, nil
}
```

## Configuration Access

### API Interface

```go
type ConfigAPI struct {
    // Get compiled config for current tenant
    router.Get("/api/config", api.GetCurrentConfig)
    
    // Get raw config for current tenant
    router.Get("/api/config/raw", api.GetRawConfig)
    
    // Update config for current tenant
    router.Put("/api/config", api.UpdateConfig)
    
    // View inheritance chain
    router.Get("/api/config/inheritance", api.GetInheritanceChain)
    
    // Get groups for current tenant
    router.Get("/api/groups", api.GetGroups)
    
    // Get inherited groups for current tenant
    router.Get("/api/groups/inherited", api.GetInheritedGroups)
}
```

### CLI Commands

```bash
# View compiled config
cfgms config get

# View raw config
cfgms config get --raw

# Update config
cfgms config set <path> <value>

# View inheritance
cfgms config inheritance

# View groups
cfgms groups list

# View inherited groups
cfgms groups inherited
```

## Best Practices

1. **Minimal Overrides**
   - Only override what needs to be different
   - Keep inherited values when possible

2. **Group Management**
   - Use public groups for shared configurations
   - Use private groups for tenant-specific configurations
   - Keep group definitions simple and focused

3. **Documentation**
   - Document why overrides are needed
   - Maintain change history

4. **Validation**
   - Validate configs before applying
   - Ensure compatibility with parent configs

5. **Testing**
   - Test inheritance chains
   - Verify override behavior
   - Test group membership
