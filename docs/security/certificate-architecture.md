# Certificate Architecture

## Overview

CFGMS uses **separated certificate architecture** (mandatory as of Issue #1765). Every
certificate is minted for exactly one declared purpose. There is no unified mode.

## Purpose Model

The `CertificatePurpose` enum in `pkg/cert/types.go` is the authoritative selector.
Call `GetCurrentCertForPurpose(P)` (presentation/signing paths) or
`GetAllValidCertificatesForPurpose(P)` (verification/trust paths). Do not call
`GetCertificatesByType` directly in transport or TLS hot paths.

| Purpose | `CertificatePurpose` | Underlying `CertificateType` | Who mints | Who consumes | Lifetime | Singleton |
|---------|----------------------|------------------------------|-----------|--------------|----------|-----------|
| gRPC-over-QUIC mTLS | `PurposeTransport` | `CertificateTypeInternalServer` (4) | Controller CA | Controller + steward mTLS handshake | 365 days | Yes |
| HTTP REST API | `PurposeAPI` | `CertificateTypePublicAPI` (3) | Controller CA (internal) or external (certbot) | HTTPS clients, browser | 365 days (internal) / operator-managed (external) | Yes |
| Config/DNA signing | `PurposeSigning` | `CertificateTypeConfigSigning` (5) | Controller CA | Steward verifies signed configs | 1095 days (3 years) | Yes |
| Steward mTLS client | `PurposeClient` | `CertificateTypeClient` (2) | Controller CA | Controller authenticates stewards | 365 days | No (one per steward) |

## Certificate Properties

### Transport (`PurposeTransport` → `CertificateTypeInternalServer`, type=4)

```
EKU:      ServerAuth
KeyUsage: KeyEncipherment, DigitalSignature
Key:      2048-bit RSA (default)
Lifetime: 365 days
SANs:     localhost, cfgms-internal, controller-standalone, 127.0.0.1, operator-configured
```

Generated by `EnsureSeparatedCertificates` on first boot (idempotent).

### Public API (`PurposeAPI` → `CertificateTypePublicAPI`, type=3)

```
EKU:      ServerAuth
KeyUsage: KeyEncipherment, DigitalSignature
Key:      2048-bit RSA (default)
Lifetime: 365 days (internal) or operator-managed (external)
```

Two sources:
- **Internal** (default): generated by `GenerateServerCertificate` on demand when
  `GetCurrentCertForPurpose(PurposeAPI)` finds no cert.
- **External**: loaded from `cert_path`/`key_path` (e.g., certbot/Let's Encrypt).
  Set `CFGMS_CERT_PUBLIC_API_SOURCE=external`.

### Signing (`PurposeSigning` → `CertificateTypeConfigSigning`, type=5)

```
EKU:      CodeSigning ONLY (no ServerAuth)
KeyUsage: DigitalSignature ONLY (no KeyEncipherment)
Key:      4096-bit RSA
Lifetime: 1095 days (3 years)
```

Generated by `EnsureSeparatedCertificates` on first boot (idempotent). Stewards cache
this cert at registration and reject configs signed by a different key — it must remain
byte-identical across controller restarts.

### Client (`PurposeClient` → `CertificateTypeClient`, type=2)

```
EKU:      ClientAuth
KeyUsage: DigitalSignature
Key:      2048-bit RSA (default)
Lifetime: 365 days (default)
```

One certificate per steward, issued at registration.

## Type Enum Stability

Certificate type values are explicit integers (not iota) to prevent stored `metadata.json`
corruption if types are reordered:

```go
CertificateTypeCA              = 0
// Integer value 1 is reserved (formerly CertificateTypeServer, removed in Issue #1765)
CertificateTypeClient          = 2
CertificateTypePublicAPI       = 3
CertificateTypeInternalServer  = 4
CertificateTypeConfigSigning   = 5
```

## First-Boot Generation

`EnsureSeparatedCertificates(internalCfg, signingCfg)` is called on every controller
startup (both `--init` and normal boot). It is a true no-op when certs already exist —
no new cert is minted, no serial changes. This guarantees restart stability (AC5).

The public API cert is **not** generated by `EnsureSeparatedCertificates`. It is created
on demand by `getServerCertificate()` when the HTTP server first starts.

## Migrating from Unified Mode

### Why you might see this error

```
startup blocked: found N legacy unified-mode certificate(s) (type=1,
formerly CertificateTypeServer) in the certificate store
```

Your certificate store contains certs with integer type 1 (`CertificateTypeServer`), which
was removed in Issue #1765. This is a **hard break**: unified mode is not supported. No
silent dual-identity is possible.

### Remediation (development/pre-GA installs)

1. Stop the controller.
2. Wipe the certificate store:
   ```bash
   rm -rf /var/lib/cfgms/certs   # or your configured CAPath
   ```
3. Re-run initialization:
   ```bash
   controller --init
   ```
4. Start the controller normally.

All stewards must re-register after the CA is wiped, since the signing certificate will
have a new key.

### Why there is no migration command

The signing certificate is the steward's trust anchor. Migrating from unified mode
(where `CertificateTypeServer` was used for signing) to separated mode (dedicated
`CertificateTypeConfigSigning`) requires re-establishing trust with all stewards. There
is no safe online migration path. Wipe and re-init is the correct procedure.

## Registration Response

The registration response always includes the dedicated signing certificate:

```json
{
  "server_cert": "<signing cert PEM (backward compat alias)>",
  "signing_cert": "<dedicated signing cert PEM>"
}
```

Stewards prefer `signing_cert` when present; `server_cert` is kept for backward compat
with older stewards.

## External Public API Certificate

```bash
CFGMS_CERT_PUBLIC_API_SOURCE=external
CFGMS_CERT_PUBLIC_API_CERT_PATH=/etc/letsencrypt/live/api.example.com/fullchain.pem
CFGMS_CERT_PUBLIC_API_KEY_PATH=/etc/letsencrypt/live/api.example.com/privkey.pem
```

The controller loads these files on startup. Renewal (certbot auto-renew) takes effect on
the next controller restart or when the HTTP server reloads its TLS config.
