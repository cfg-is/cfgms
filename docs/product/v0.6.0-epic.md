# Epic: v0.6.0 - Finalizing Foundational Endpoint CMS

**Epic ID**: v0.6.0
**Type**: Major Milestone / Release Epic
**Status**: 📋 Planned
**Story Points**: 65 points (5 stories)
**Priority**: HIGH - Completes foundational endpoint CMS capabilities
**Created**: 2025-10-10

---

## Executive Summary

v0.6.0 completes the foundational endpoint configuration management system by implementing advanced script execution, compliance-focused configuration templates, enterprise-grade Windows patching with major version upgrades, and production-ready performance monitoring for both endpoints and controller infrastructure. This release transforms CFGMS from core platform (v0.5.0) into a fully-featured, production-ready endpoint configuration management system capable of enterprise-scale MSP operations with CMMC/CIS compliance support.

**Core Value Proposition**: Policy-driven automation where MSPs define configurations declaratively and CFGMS continuously enforces them automatically.

---

## Epic Goal

Deliver production-ready endpoint management capabilities that enable MSPs to:
1. Execute scripts safely across multi-platform fleets with DNA-based parameterization
2. Apply CIS/CMMC compliance templates automatically
3. Manage Windows patching (including Win 10→11 upgrades) via declarative policy
4. Monitor endpoint and controller performance proactively
5. Maintain 99.9%+ system availability with automated performance management

**Success Metric**: MSPs can achieve CMMC Level 1 or CIS Level 1 compliance across their fleet using CFGMS automation alone.

- Batch/cmd (.bat, .cmd) - Windows legacy

**4. Workflow Engine Integration**
- Immediate execution via workflow trigger

- Complex orchestration (pre-checks → execution → validation → rollback)

**5. Real-Time Execution Monitoring**

- Capture and return exit codes to workflow engine

- No automatic retry logic (admin-controlled via workflows)

**8. Workflow Integration Patterns**
- Execute script as workflow step
- Pass workflow variables to script parameters
- Script output becomes workflow variables
- Error handling and rollback via workflow engine

#### How It Integrates

- Leverages DNA system (Story 78) for parameter injection
- Integrates with workflow engine (Stories 2.1-2.6) for orchestration

### Story 3: Windows Patch Compliance (Advanced) (13 points)

#### What's Being Added

**1. Policy-Driven Patch Management Engine**

**Declarative Patching Policy** (device/group/client/MSP config):
```yaml
windows_patching_policy:
  patch_types:
    critical:
      install_after_days: 7    # Install 7 days after Microsoft releases
      max_compliance_days: 14  # Must be installed within 14 days
    important:
      install_after_days: 14
      max_compliance_days: 30
    moderate:
      install_after_days: 30
      max_compliance_days: 60
    optional:
      install_after_days: 90
      max_compliance_days: 180

  feature_updates:
    target_version: "Windows 11 24H2"
    install_after_release_days: 90
    compatibility_checks: true

  maintenance_windows:
    - day: sunday
      start: "02:00"
      end: "05:00"
    honor_user_presence: true
    max_defer_days: 7

  compliance_alerting:
    warning_days_before_breach: 7
    critical_days_before_breach: 1
```

**CFGMS Enforcement**:
- Continuously monitors patch release dates (Microsoft Update Catalog)
- Automatically installs patches when `install_after_days` + maintenance window
- Proactively alerts when approaching `max_compliance_days`
- Enforces policy across all devices without manual intervention

**2. Windows Update COM API Integration**
- `IUpdateSession`, `IUpdateSearcher`, `IUpdateDownloader`, `IUpdateInstaller` COM interfaces
- Go ↔ COM interop via `github.com/go-ole/go-ole`
- Search, download, install patches locally on endpoints
- Filter patches by severity and release date

**3. Major Version Upgrade Policy Enforcement**

**Policy-Driven Upgrades**:
```yaml
feature_updates:
  target_version: "Windows 11 24H2"
  install_after_release_days: 90
  compatibility_checks: true
```

**CFGMS Behavior**:
1. Validates hardware compatibility (TPM 2.0, CPU, RAM, disk)
2. Waits 90 days after Microsoft releases 24H2
3. Automatically upgrades during maintenance window
4. Monitors DNA post-upgrade for validation
5. Reports completion/failure (Windows built-in rollback handles failures)

**Compatibility Validation** (DNA-based):
- TPM 2.0 present
- UEFI boot enabled
- CPU on Windows 11 compatibility list
- RAM ≥4GB
- Free disk space ≥64GB

**4. Maintenance Window Enforcement** (Generic System Feature)

**Check if already implemented** - If not, build as:
```yaml
# Cascading configuration (MSP → Client → Group → Device)
maintenance_windows:
  reboot_allowed:
    - day: sunday
      start: "02:00"
      end: "05:00"
    - day: wednesday
      start: "02:00"
      end: "04:00"

  honor_user_presence: true
  max_defer_days: 7
  emergency_override: false
```

**Enforcement**:
- ALL reboot operations check maintenance window (not just patching)
- Modules call `device.CanReboot()` before rebooting
- Defers reboot if outside window (up to `max_defer_days`)
- Forces reboot after `max_defer_days`

**5. Proactive Compliance Alerting**

**Alert Examples**:
- **Warning**: "Device DESKTOP-123 will breach patch compliance in 5 days (Critical patch KB5034441 not installed)"
- **Critical**: "Device SERVER-456 will breach compliance in 1 day (10 Important patches missing)"
- **Breach**: "Device LAPTOP-789 is NON-COMPLIANT (Critical patch KB5034441 not installed - 3 days overdue)"

**Alert Delivery**:
- CFGMS dashboard (when built)
- Workflow engine triggers (email, webhooks)
- MSP admin notification queue
- Per-tenant alert configuration

**6. Basic Compliance Reporting + Alerting**

**Report Contents**:
- Device name, OS version, patch level
- Missing patches (count by severity: Critical/Important/Moderate)
- Last patch date
- Compliance status: ✅ Compliant / ⚠️ Warning / 🔴 Non-Compliant
- Win 11 upgrade eligibility (Yes/No/Warning)
- Days until compliance breach (countdown)

#### How It Integrates
- Uses Windows Update COM API for local patch management
- Leverages workflow engine for orchestration
- Uses DNA system for compatibility validation
- Integrates with configuration rollback (Story 75)
- Uses multi-tenant isolation (Story 12.5)
- Integrates with reporting framework (Story 5.1)
- Uses pluggable storage for patch policy

#### Success Criteria
- [ ] MSPs define patch policy declaratively in configuration
- [ ] CFGMS automatically installs patches per policy
- [ ] Maintenance windows honored for all reboot operations (not just patching)
- [ ] Proactive alerting 7 days before compliance breach
- [ ] Critical alerting 1 day before compliance breach
- [ ] Major version upgrades (Win 10→11, 23H2→24H2) policy-driven
- [ ] Hardware compatibility validation blocks incompatible upgrades
- [ ] Basic compliance reporting shows device status + days until breach
- [ ] Windows Update COM API functional on all Windows versions

### Story 4: Endpoint Performance Monitoring (13 points)

#### What's Being Added

**1. Performance Metrics Collection** (Separate from DNA)

**DNA System** (existing - unchanged):
- Purpose: System "blueprint" (hardware specs, installed software)
- Change rate: Low (only during upgrades/installs)
- Storage: Relational database (Git/PostgreSQL/SQLite)
- Collection: Every 5 minutes (configurable)

**Performance Monitoring** (new):
- Purpose: System "vital signs" (resource utilization)
- Change rate: High (changes every second)
- Storage: Time-series database (InfluxDB/TimescaleDB)
- Collection: Every 60 seconds (configurable)

**Basic Resource Metrics**:
```yaml
performance_metrics:
  cpu:
    usage_percent: 73.2      # Overall CPU utilization

  memory:
    total_bytes: 17179869184
    used_bytes: 8796093022
    usage_percent: 51.2

  disk:
    read_bytes_per_sec: 15728640
    write_bytes_per_sec: 5242880
    usage_percent: 68.5

  network:
    bytes_sent_per_sec: 1048576
    bytes_received_per_sec: 2097152

  system:
    online_status: true
    last_checkin: "2025-10-10T14:32:15Z"
    uptime_seconds: 259200
```

**Top Resource Consumers**:
```yaml
top_processes:
  cpu:
    - name: "chrome.exe", pid: 4521, cpu_percent: 23.5
    - name: "sqlservr.exe", pid: 1234, cpu_percent: 18.2

  memory:
    - name: "chrome.exe", pid: 4521, memory_mb: 2048
    - name: "sqlservr.exe", pid: 1234, memory_mb: 1536
```

**2. Process and Service Monitoring**

**Watchlist Configuration**:
```yaml
process_monitoring:
  watched_processes:
    - name: "sqlservr.exe"
      alert_if_missing: true
      auto_start: false
      remediation_workflow: "restart-sql-server"

  watched_services:
    - name: "MSSQLSERVER"
      alert_if_stopped: true
      auto_start: false
      remediation_workflow: "sql-service-recovery"
```

**Behavior**:
- Monitor if specific processes/services exist/running
- Alert if missing/stopped
- Optionally attempt to start (if `auto_start: true`)
- Trigger workflow for complex remediation

**3. Threshold-Based Alerting**

**Alert Configuration**:
```yaml
performance_alerting:
  thresholds:
    cpu:
      warning: 80
      critical: 95
    memory:
      warning: 85
      critical: 95
    disk:
      warning: 90
      critical: 98

  alert_durations:
    warning: 300   # 5 minutes sustained
    critical: 120  # 2 minutes sustained
```

**4. Time-Series Storage Architecture**

**Separate Storage**:
- **DNA Storage**: Git/PostgreSQL/SQLite (point-in-time snapshots)

- **Retention**: 30 days (default, configurable at global/MSP/company level)

**5. Workflow-Driven Remediation**

No automatic remediation - trigger workflows instead:
```yaml

  steps:

```

**Collection Methods**:
- **Windows**: PowerShell (`Get-Counter`, `Get-Process`)
- **Linux**: `/proc` filesystem (`/proc/stat`, `/proc/meminfo`)
- **macOS**: `sysctl`, `vm_stat`, `iostat`

**Implementation**:

### Story 5: Server/Controller Performance Monitoring (13 points)

#### What's Being Added

**1. Controller Performance Metrics Collection**

**MQTT Broker Metrics** (CRITICAL):
- Active steward connections
- Connection pool saturation
- Message queue depth (pending messages)
- Message throughput (msgs/sec)
- Message publish-to-delivery latency

**Storage/Database Metrics** (CRITICAL):
- Connection pool utilization
- Query latency (p50, p95, p99)
- Slow query count (>1 second)
- Transaction throughput
- Lock contention

**Application Metrics** (CRITICAL):
- **Workflow queue depth** (waiting workflows)
- **Script execution queue depth** (waiting scripts)
- Active workflow executions
- Configuration updates in progress
- DNA collections per minute

**System Metrics** (per controller node):
- CPU utilization
- Memory usage (heap, RSS)
- GC pause times
- Goroutine count
- File descriptor usage

**Collection Interval**: 10 seconds

**2. Proactive MSP Impact Detection**

**Key Performance Indicators**:
```yaml
queue_metrics:
  workflow_queue:
    depth: 45              # 45 workflows waiting
    avg_wait_time_sec: 12  # 12 seconds average wait
    max_wait_time_sec: 45  # Oldest workflow waiting 45 seconds

  script_queue:
    depth: 23
    avg_wait_time_sec: 8
    max_wait_time_sec: 30
```

**Alert Triggers**:
- ⚠️ **WARNING**: Queue depth >20 OR avg wait >10 seconds
- 🔴 **CRITICAL**: Queue depth >50 OR avg wait >30 seconds
- 🚨 **EMERGENCY**: Queue depth >100 OR wait >60 seconds

**Detection Within 30 Seconds**: Proactive alerts BEFORE MSPs notice delays

**3. MSP Impact Classification**

**CRITICAL**: Steward registrations failing, config delivery blocked, module execution failing
**HIGH**: Operations delayed >30 seconds (workflow/script queues)
**MEDIUM**: Non-critical slowness (reporting, admin operations)

**4. Automated Self-Healing** (Conservative)

**Safe Automatic Actions**:
- ✅ Expand MQTT connection pool
- ✅ Expand database connection pool
- ✅ Warm frequently accessed caches
- ✅ Trigger garbage collection
- ✅ Defer non-critical operations (reporting, historical queries)

**Admin-Approval Required**:
- ⚠️ Close idle connections
- ⚠️ Throttle background tasks
- ⚠️ Restart unhealthy background workers
- ❌ Restart controller node

**5. Admin-Approved Tenant Migration**

**Automatic Detection + Manual Approval**:
1. System detects overloaded controller node (CPU >90%, queue depth >50)
2. System analyzes tenant distribution across cluster
3. System **recommends** tenants to migrate (smallest/least active first)
4. **Admin approves** migration via CLI/API
5. System executes graceful migration (<60 seconds):
   - Pause new operations for tenant
   - Complete in-flight operations
   - Update MQTT routing (stewards reconnect to new node)
   - Resume operations on new node
6. Validate success, rollback if failure

**6. Circuit Breaker Patterns**

**Database Circuit Breaker**:
- 5 failures in 10 seconds → OPEN circuit
- Queue writes, serve reads from cache
- Retry every 30 seconds

**External API Circuit Breaker**:
- 3 failures → OPEN circuit
- Defer SaaS operations
- Retry every 60 seconds

**MQTT Broker Circuit Breaker**:
- Broker unresponsive → OPEN circuit
- Queue outbound messages
- Failover to backup broker (if HA)

**7. Email-Based Alert Escalation**

**Tier 1 - Warning**: Audit log only
**Tier 2 - Critical**: Email admins with summary + recommended actions
**Tier 3 - Emergency**: Email escalation list with runbook

**Email Template**:

## Risk Assessment & Mitigation

### High Risk: Windows Major Version Upgrades (Story 3)

**Risk**: Major upgrades (Win 10→11) could fail and brick devices
**Impact**: Critical - Device unusable, requires manual intervention
**Probability**: Medium (10-15% failure rate in enterprise)

**Mitigation**:
- DNA-based compatibility validation blocks incompatible devices
- Windows built-in rollback handles failures (<10 minutes recovery)
- Phased rollout via device groups (test → pilot → production)
- Automated pre-upgrade backups via workflow
- Emergency stop mechanism (pause all upgrades)

**Rollback Plan**:
- Windows Recovery Environment automatic rollback
- Maximum 10-minute recovery time (Windows built-in)
- Configuration snapshot restoration via git

---

### Medium Risk: Controller Performance Impact (Story 5)

**Risk**: Performance monitoring overhead degrades controller
**Impact**: High - Could affect all stewards (50k+ devices)
**Probability**: Low (5% if not optimized)

**Mitigation**:
- Metrics collection optimized for <1% CPU overhead
- Sampling strategies for high-cardinality metrics
- Circuit breakers prevent cascading failures
- Feature flag to disable monitoring if needed

**Rollback Plan**:
- Disable performance monitoring via config
- Fall back to Story 7.1 basic monitoring
- No impact on core controller functionality

---

### Medium Risk: Script Execution Security (Story 1)

**Risk**: Malicious scripts could compromise endpoints
**Impact**: Critical - Full device compromise
**Probability**: Low (with proper controls)

**Mitigation**:
- RBAC-based script approval workflow
- Audit logging for all script executions
- Script signature verification (future: code signing)
- Multi-tenant isolation (tenant scripts cannot cross boundaries)

**Rollback Plan**:
- Kill running scripts via MQTT command
- Revert configuration to pre-script state
- Quarantine compromised devices
- Comprehensive audit trail for forensics

---

### Low Risk: Template Complexity (Story 2)

**Risk**: Compliance templates too complex, leading to errors
**Impact**: Medium - Configuration errors could disable services
**Probability**: Medium (30-40% initial misconfiguration)

**Mitigation**:
- Community-tested templates from marketplace
- Template validation framework catches errors
- DNA-based conditional logic handles edge cases
- Configuration rollback (Story 75) handles failures

**Rollback Plan**:
- Git-based version control (instant revert)
- Automatic rollback on validation failure
- Manual rollback via cfgctl CLI
